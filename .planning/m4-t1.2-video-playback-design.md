## 1. Context & why this is one of three sub-projects

The end goal the user set is **A/V-synced playback in Shaka MSE from a real
FFmpeg `moq_mmt` MMTP multicast stream**. That goal spans three subsystems
across three repos and is too large for a single spec, so it is decomposed:

| Sub-project | Scope | Repos | Depends on |
|---|---|---|---|
| **A — Video playback (THIS spec)** | reassembled video MFU → transmux → MSE; video frames render | shaka-player | — (unblocked now) |
| **B — Publisher audio emission** | FFmpeg `moq_mmt` emits an audio MMTP track; `moq-pub-mmtp` publishes it; catalog advertises it | FFmpeg fork, moq-rs | — |
| **C — Receiver audio + A/V sync** | reassemble audio MFU → transmux → audio SourceBuffer; shared sync reference | shaka-player | A + B |

**Build order: A → B → C.** A is independently provable against today's
**video-only** stream (confirmed: `moq-pub-mmtp/src` has zero audio code; the
T1.7 capture fixtures are Init + video MFU only). A also de-risks the
transmux/MSE/timing machinery that C reuses. Each sub-project gets its own
spec → plan → implementation cycle.

### Decision record
- **Scope of A:** *full video playback* — frames render end-to-end in the
  live/Karma MSE harness, provable against the current real stream. (Audio +
  A/V sync deliberately deferred to B/C.)
- **Transmux approach: A3 — dedicated `MmtpTransmuxer`** (chosen over A1
  "delegate to `LocTransmuxer`" and A2 "catalog init + media-only synth"). A3
  front-loads MMTP-awareness (FEC trailer tolerance, explicit timing input, a
  sync seam for C) and keeps MMTP concerns out of `LocTransmuxer`, at the cost
  of slightly more code now. This realizes the recorded D3 "thin-delegate vs
  dedicated" decision in favor of dedicated.

## 2. Ground truth (from T1.7; memory `ffmpeg-moq-mmt-multicast-wire-format.md`)

- Multicast leg emits **Init (FT=0)** = `MMTP+MPU+(ftyp+mmpu+moov)` and
  **MFU (FT=2)** only. FT=1/moof never on wire.
- **MFU payload = AVCC length-prefixed NAL** (`[4B len][NAL]`, SEI `nal_type=6`
  then coded NALs) — **H.264**. Not fMP4 → media segments must be *synthesized*.
- Init bytes == catalog `track.initData` (`moq-catalog/src/lib.rs:49`); Shaka
  already reads it at `msf_parser.js:685`. It is `ftyp+**mmpu**+moov` (an MMT box
  between ftyp and moov) → not guaranteed MSE-clean as-is.
- **Timing:** MMTP `timestamp` = `us_to_ntp_short(base_pts + i·sample_dur)`,
  constant across a frame's fragments, distinct/monotonic per frame, frame 0 = 0
  (→ never use raw timestamp as subgroup_id; map to per-group index ≥ 1).
- Test stream = testsrc2 15fps GOP15, **no B-frames → PTS = DTS**.
- `mpu_sequence` = MoQ group_id (increments per keyframe → group = MPU = GOP).
- FEC `SourceFecPayloadId` is a **trailer** past MPU `payload_length`, but
  `moq-pub-mmtp` + `mmtp_parser.js` model it as a prefix — a parser-parity gap
  owned by **T1.FEC**; A must at least tolerate/skip it.

## 3. Existing wiring being mirrored (shaka-player)

- `msf_parser.js processTrack_` (~681) branches `isLoC`/`isMmtp` on
  `track.packaging` + `selectionParams.codec` (codec read fixed in `24d0d727b`).
- The **LOC branch is the template**: `createSegmentIndex` →
  `subscribeToTrack(track, key, (obj) => …)` → derive `{startTime, duration,
  payload}` → `new SegmentReference(...)` + `reference.setSegmentData(payload)` →
  `segmentIndex.merge(...)` → `presentationTimeline.notifySegments(...)`.
  LOC derives frame duration from catalog via
  `LOCParser.frameDurationFromTrack(track)` (no dependence on optional
  per-object timestamps).
- **NAL→fMP4 conversion runs in `MediaSourceEngine` / `TransmuxerEngine` at
  append time** via a registered `shaka.extern.Transmuxer` (`LocTransmuxer`,
  `ts_transmuxer`, …). This is the extension slot `MmtpTransmuxer` plugs into.
- Current MMTP path is **observe-only**: `processTrack_:701` → `observeMmtpTrack_`
  → `MmtpTrackProcessor.process()` → log records. `MmtpTrackProcessor`/
  `MfuReassembler` already emit `'mfu'` records (complete frame NAL + group,
  subgroup, rapFlag, timestamp, mpuSequence). These are **reused unchanged**.

## 4. Design

### 4.1 Components
- **`shaka.transmuxer.MmtpTransmuxer`** — new, `lib/transmuxer/mmtp_transmuxer.js`.
  A `shaka.extern.Transmuxer` registered in `TransmuxerEngine`, keyed off a
  `moq/mmt` mimeType. Modeled on `LocTransmuxer`: consumes a reassembled MFU
  (one frame's AVCC NAL), runs `H264.parseInfo()` for SPS/PPS, emits fMP4 (init
  once + moof/mdat per frame) via `Mp4Generator`. MMTP-specific: tolerates the
  FEC trailer; takes frame **timing as an explicit input** (does not self-count)
  — the startTime/duration computed by `processMmtpTrack_` arrives via the
  `SegmentReference` passed to the `shaka.extern.Transmuxer` `transform()` call
  at append time; exposes a timing/sync seam for Sub-project C. Shares the
  `H264`/`Mp4Generator` helpers with `LocTransmuxer` (no duplication).
- **`processMmtpTrack_`** in `msf_parser.js` — replaces `observeMmtpTrack_`; an
  `isMmtp` branch parallel to `isLoC`. Builds the `shaka.extern.Stream` with
  `mimeType = moq/mmt; codecs="<selectionParams.codec>"`, derives per-frame
  duration from catalog framerate/timescale, and in the `createSegmentIndex`
  `subscribeToTrack` callback feeds objects to the reassembler, emitting one
  `SegmentReference` per completed MFU.
- **Reused unchanged:** `MmtpTrackProcessor` / `MfuReassembler`,
  `TransmuxerEngine`, `MediaSourceEngine`, `SegmentReference` / `InitSegmentReference`,
  `presentationTimeline`.

### 4.2 Data flow
`subscribeToTrack(obj)` → `MmtpTrackProcessor.process(obj.data, obj.location)`:
- **`'mfu'`** (complete frame) → derive `startTime`/`duration` (§4.3) →
  `SegmentReference` with `setSegmentData(nalFrame)` + `moq/mmt` mimeType →
  `segmentIndex.merge` → `presentationTimeline.notifySegments`. At MSE append,
  `TransmuxerEngine` routes the segment through `MmtpTransmuxer` → fMP4 →
  SourceBuffer.
- **`'init'`** → ignored for playback (init synthesized by transmuxer; catalog
  `initData` is the fallback — see §4.3 open item).
- **`'repair'` / `'non-media'`** → dropped (T1.FEC owns policy).
- **incomplete fragment** → `null` (buffered, bounded by the reassembler cap).
- **`FragmentType.FRAGMENT`** → never on wire → **error**.

### 4.3 Timing & init
- **Duration:** derived from catalog framerate/timescale (an MMTP analog of
  `LOCParser.frameDurationFromTrack`). **No magic numbers; throw if absent** for
  an mmtp track.
- **startTime:** accumulated per frame in presentation order (group/MPU asc →
  subgroup/MFU asc). **PTS = DTS** (no B-frames in the stream); B-frame reorder
  is a documented non-goal for A, deferred to the H264 parse path if it ever
  appears.
- **Init / mid-stream join:** stream is live; publisher re-sends Init and starts
  each group with an IDR carrying SPS/PPS. `MmtpTransmuxer` builds the init
  segment from the first IDR's SPS/PPS; media before the first IDR is dropped
  (bounded).
- **OPEN implementation item (settle in plan):** init from transmuxer-synthesized
  SPS/PPS **vs** catalog `track.initData` (which is `ftyp+mmpu+moov` — may need
  `mmpu` stripped for MSE). **Default: transmuxer-synthesized** (sidesteps the
  `mmpu`/MSE question); keep catalog init as a fallback.

### 4.4 Error handling (error-on-bad-init; no silent defaults)
- Missing catalog framerate/timescale for an mmtp track → **throw**.
- `FragmentType.FRAGMENT` seen → **throw** (wire-contract violation).
- Reassembly memory bounded by the existing **required** `maxBufferedMfus` cap
  (no timers).
- FEC trailer must not corrupt the parse (parity gap owned by T1.FEC; A
  tolerates/skips it).

### 4.5 Testing
- **Unit:** `MmtpTransmuxer` over reassembled-NAL fixtures → assert valid fMP4
  (ftyp+moov init, moof+mdat media), reusing the Rust→JSON parity-harness pattern
  or a JS golden.
- **Integration (Karma):** extend the live observe-dump harness (`d88d3a3c2`)
  from "dump records" to "append to MSE + assert `buffered.length > 0` and
  `currentTime` advances."
- **E2E:** extend `.planning/m4-t1.7-e2e.sh` — real `moq_mmt` → relay → Shaka →
  **frames render** (buffered range grows, playback progresses).

### 4.6 Sub-project C seam (why A3 earns its keep now)
`MmtpTransmuxer` takes timing as explicit input and isolates the per-track media
timeline, so C can add a shared audio↔video sync reference without reworking the
video path. The `moq/mmt` mimeType + `TransmuxerEngine` registration generalize
to an audio codec.

## 5. Out of scope (explicit)
- Audio of any kind; A/V sync (→ C).

- FEC repair policy / parser-parity fix (→ T1.FEC).
- Publisher / FFmpeg changes (→ B).
- Per-track history-window override (already noted future work in B-MIG-pub).
