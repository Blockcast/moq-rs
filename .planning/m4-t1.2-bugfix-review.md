# M.4 T1.2 — two-stage review of the bundled "render-flow" fixes (2026-05-31)

**Scope:** the three production-code changes bundled into shaka-player commit
`4df077593` (`test(e2e): T1.2 harness plays MMTP video + asserts playback`).
The handoff (`m4-t1.2-session-handoff.md`, caveat #3) flagged these as UNREVIEWED
production changes shipped inside a `test(e2e):` commit. This doc is the review.

Reviewed against: design `m4-t1.2-video-playback-design.md` (decision record +
§2 ground truth), wire-format memory `ffmpeg-moq-mmt-multicast-wire-format.md`,
and the repo architectural principles in `CLAUDE.md` ("no silent fallbacks",
"error on bad init", scope discipline, dead-code hygiene).

The three changes:
1. `lib/msf/mmtp_track_processor.js` — strip the 14-byte MFU sub-header off the first fragment of timed MFUs.
2. `lib/msf/msf_parser.js` — `processMmtpTrack_`: on `kind==='init'`, `setSegmentData(InitMPU)` on the initSegmentReference.
3. `lib/transmuxer/mmtp_transmuxer.js` — new `seedFromFmp4Init()` (86 lines).

## Stage 1 — spec conformance

### The central finding (F-X2, HIGH): D9 contradicts its own ground truth
- **D9** (design decision record): *"init is synthesized from in-band SPS/PPS
  (LocTransmuxer pattern), not the catalog `moov`. The mmpu-strip task is
  DROPPED. No catalog init, no fallback."* It supersedes **D-init** (which had
  `moov` as the primary init, transmuxer media-only).
- **§2 Ground truth, line 42** of the *same design doc*: *"MFU payload = AVCC
  length-prefixed NAL (H.264; SEI `nal_type=6` then coded NALs)."* Line 43:
  *"Init bytes == catalog `track.initData`"* (`ftyp+mmpu+moov`).
- Wire-format memory confirms: SPS/PPS live in the `moov`/`avcC` (the Init MPU);
  MFUs carry SEI + coded slices, **not parameter sets**.
- ⇒ D9's premise ("first IDR's in-band SPS/PPS") is **false for the `moq_mmt`
  publisher**. LocTransmuxer's in-band-SPS/PPS pattern fits LOC packaging, not
  MMTP. The original **D-init was correct**; D9 reverted to the wrong pattern.
- This is caveat #1 of the handoff, now nailed with in-doc citations.
- **Recommendation:** amend the decision record — **D12: revert D9, reinstate
  D-init.** Init = the Init-MPU `moov` (mmpu-stripped → clean `ftyp+moov`, or
  seed the transmuxer's avcC via `seedFromFmp4Init`); transmuxer media-only. This
  is the init-path rewrite to green-light next; it subsumes #2 and #3.

### Change #1 — conforms; fills a plan gap
- Matches §2 ground truth and the memory (the 14B MFU DU header — `mfs,
  sample_number, offset, prio, dep` — is written **only on the first fragment**,
  FFmpeg `moqenc_mmt.c:4393`).
- Not anticipated by any decision: the plan assumed `MfuReassembler` emits clean
  AVCC NAL. The first-fragment DU header was an unaccounted-for wire detail.
  So #1 is a **correct fix for a real defect the plan overlooked** — legitimately
  in scope for "make the real stream play."

### Change #2 — violates D9 and the no-fallback principle
- `processMmtpTrack_` builds `initSegmentReference` with **empty** initData
  (`msf_parser.js:631-639`), commented *"the transmuxer synthesizes the real init
  segment from the first IDR's in-band SPS/PPS (decision D9)."*
- Fix #2 then `setSegmentData(InitMPU)` (`:678-688`), commented *"the Init MPU is
  the canonical source; D9 ... serves as a fallback."*
- These two comments **directly contradict** (~40 lines apart) and encode a
  silent fallback — banned by D9 ("no fallback") AND `CLAUDE.md` ("no silent
  fallbacks"). It is the wrong *shape* of the right idea (#2 is right that the
  Init MPU is canonical — which is exactly why D9 must be reverted, not patched
  with a fallback).

### Change #3 — unwired, but points the right way
- Not called anywhere ⇒ no runtime spec conformance. Its mere existence is an
  implicit admission that the avcC/Init-MPU path (D-init) is the real one.

## Stage 2 — code quality

### Change #1 — `mmtp_track_processor.js`
- **F-1.1 (LOW):** gate is `isFirstFragment && mpu.timed`. The publisher's actual
  emit condition is `is_first` alone (FFmpeg `:4393`). Harmless here (stream is
  always timed) but the comment over-claims ISO precision about non-timed MFUs
  this stream never produces. Either drop `&& mpu.timed` to match the publisher,
  or document the gate as defensive-given-always-timed.
- **F-1.2 (MEDIUM):** **no unit test.** `test/msf/mmtp_track_processor_unit.js`
  exists but doesn't cover the strip; only the E2E exercises it. Design §4.7
  mandates unit coverage for processor changes. Add: first-frag DU stripped,
  continuation frags (FI=2/3) untouched, non-timed untouched, short-payload guard.
- **F-1.3 (LOW):** the class docstring (`:13-27`) still says this is the "T1.5a
  observe-first stage… does NOT build CMAF or touch MSE… playback wiring lands
  after T1.7." That's now stale — the processor performs wire-level AU extraction
  feeding playback. Update the docstring.

### Change #2 — `msf_parser.js`
- **F-2.1 (HIGH):** dual unreconciled init path (empty-initData vs stored
  Init-MPU); which one MSE consumes is ambiguous. The production player path is
  unproven and deadlocks (handoff caveat #2).
- **F-2.2 (HIGH, needs-verify):** the stored bytes are `parsed.payload` of the
  Init MPU = **`ftyp+mmpu+moov`** per §2 ground truth — i.e. **includes the
  `mmpu` box**. D-init explicitly required *stripping* `mmpu` to get a clean
  `ftyp+moov` for MSE. If anything feeds these raw bytes to a SourceBuffer, the
  `mmpu` box likely breaks the init segment. The harness sidesteps this by
  extracting `avcC` and prepending SPS/PPS — it never feeds the raw payload to
  MSE — so this latent bug is currently un-exercised. Verify before wiring.
- **F-2.3 (LOW):** no test for the init-store branch.

### Change #3 — `mmtp_transmuxer.js`
- **F-3.1 (HIGH, hygiene):** **dead code** — `seedFromFmp4Init` has zero callers;
  86 lines, untested. Per repo dead-code-hygiene + scope-discipline, it should
  not merge unwired. Either wire it in (init-path rewrite) or delete it.
- **F-3.2 (MEDIUM):** `findBox` and `findBoxRecursive` are byte-identical
  duplicates. Collapse to one.
- **F-3.3 (MEDIUM, latent bug):** the `pasp` lookup starts at `avcBody + 70`
  while `avcC` uses `avcBody + 78` (`VISUAL_SAMPLE_ENTRY_SIZE`). 70 is the *old
  wrong* VisualSampleEntry size the handoff said was corrected to 78 — a leftover.
  `pasp` would be parsed from the wrong offset (into the `avcC` box). Benign only
  because `pasp` is optional/unused-when-absent.
- **F-3.4 (POSITIVE):** `videoConfig` shape is **correct** — it extracts the
  `avcC` body (AVCDecoderConfigurationRecord, `0x01`-prefixed), matching the
  record `H264.parseInfo` constructs (`h264.js:174`). So the function is
  salvageable, not throwaway.
- **F-3.5 (MEDIUM, design):** even if seeded, `getAvcStreamInfo_` re-runs
  `H264.parseInfo(nalus)` every frame and overwrites `avcInfo_` when SPS/PPS are
  present. For `moq_mmt` (no in-band SPS/PPS) `parseInfo` returns null so the
  seed persists — but it means the MMTP transmuxer still *attempts* the in-band
  path first. Under the D9 revert, decide whether to remove the in-band attempt
  for the MMTP transmuxer entirely (single, clear init path).

### Cross-cutting
- **F-X1 (HIGH, process):** three production-code changes were bundled into a
  `test(e2e):` commit. Hides production changes in a "tests" commit and is why
  they shipped unreviewed. When acting on this review, split into properly-scoped
