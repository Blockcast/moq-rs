# T1.6 — `@blockcast/transport` WebTransport Factory (Implementation Plan / Scoping)

**Milestone:** M.4 MMTP-over-MoQ
**Status:** SCOPING (no implementation)
**Author:** scoping pass, 2026-06-01
**Scope discipline:** This is a planning doc. No source/build files touched.

---

## Summary

Each MoQ player in this repo open-codes its own WebTransport connect dance
(secure-context check, `/fingerprint` fetch + hex→bytes decode,
`serverCertificateHashes`, ALPN/subprotocol list, MoQ draft selection,
`wt.ready`/`wt.closed` handling). There are **three distinct, divergent
client-side implementations** (Shaka, hang/moq-lib, moqtail). They disagree on
ALPN lists, fingerprint-source conventions, and version policy. T1.6 proposes a
small client-side package that consolidates *only the WebTransport
establishment step* — producing a ready `WebTransport` — and leaves all
MoQ-session/SETUP logic in the players where it already lives.

**Biggest design decision (call it out up front):** the factory should return a
**raw, ready `WebTransport`** (plus the negotiated `.protocol` ALPN string), NOT
a higher-level MoQ session object. The three players each have deeply different
MoQ session layers (Shaka `MSFConnection`, moqtail `MOQtailClient`, hang
`Ietf.Connection`/`Lite.Connection`). A session-level factory would have to
absorb all three SETUP encoders and version state machines — that is a rewrite,
not a de-dup. Returning a raw WT eliminates the genuinely-duplicated 40–60 lines
(fingerprint + hashes + ALPN + secure-context) without entangling the factory in
draft-skew SETUP logic. **VERIFIED**: the duplicated region in all three ends
exactly at "we have a `wt.ready` WebTransport and an ALPN string"; everything
after diverges.

## Greenfield vs existing

### `@blockcast/transport` already exists — but it is NOT a WebTransport factory.

**VERIFIED.** `@blockcast/transport` is the published package *name* of
`packages/ssm-transport/` (`packages/ssm-transport/package.json:2` →
`"name": "@blockcast/transport"`). It is the SSM/AMT **multicast** transport
manager (DRIAD discovery, AMT gateway, ABR controller, MMT FEC client) — the
successor to the deprecated `@blockcast/multicast`
(`packages/multicast/package.json:4`: *"DEPRECATED: use @blockcast/transport"*).
It is consumed by moqtail-ts via a `file:` link
(`moqtail-private/libs/moqtail-ts/package.json:110`).

**Decision required (open):** the name `@blockcast/transport` is taken by a
multicast package whose concern (SSM/AMT receive path) is orthogonal to a
WebTransport *connect* helper. Options:
- **(A)** New package under a distinct name, e.g. `@blockcast/moq-transport` or
  `@blockcast/webtransport`. Preferred — avoids overloading "transport".
- **(B)** Add a `webtransport` subpath export to the existing
  `@blockcast/transport` (`ssm-transport`). Rejected: pulls a browser-only WT
  helper into a package that already inlines copies to *avoid* deps
  (`ssm-transport/src/abr-controller.ts:16`), and conflates SSM-receive with
  relay-connect.

This doc assumes **(A)** and uses the working name **`@blockcast/moq-transport`**
below. The task title's "`@blockcast/transport`" is a misnomer given the existing
package; flag for the milestone owner.

### Divergences worth eliminating (VERIFIED)

1. **ALPN lists disagree.** Shaka `moq-00`/`moqt-16`; hang `lite-04/03 + d17/16/15`;
   moqtail `moqt-16` only. The factory should own one source-of-truth ordered
   ALPN list keyed by a version policy.
2. **Fingerprint conventions disagree.** hang's *legacy* path hits
   `/certificate.sha256` on `http://`; hang's *new* path + the relay convention
   (per task context) is `/fingerprint`. Shaka and moqtail take a *pre-decoded*
   value and never fetch. Factory should accept **either** raw hex/bytes **or** a
   fetch URL, and standardize on the relay's `/fingerprint` endpoint.
3. **Hex→bytes decode is copy-pasted 3×** (moqtail `player.ts:702–706` hand loop;
   hang uses `Hex.toBytes`; Shaka pushes decode to caller). One helper.
4. **Secure-context guard inconsistent.** Shaka guards `window.WebTransport`
   presence (`:94`) only for the default factory; hang/moqtail don't guard at
   all. None checks `window.isSecureContext` → confusing failures on
   `file://`/`data:`.

## Proposed factory API (minimal)

Goal: erase items 1–4 above. **Resist adding layers.** One function, one
options bag, one result.

```ts
// @blockcast/moq-transport

export type MoqDraftPolicy = "auto" | "draft-17" | "draft-16" | "draft-15" | "draft-14";

export interface FingerprintSource {
  /** Pre-decoded SHA-256 (32 bytes) or hex string ("aa:bb.." or "aabb..") */
  value?: Uint8Array | string;
  /** OR fetch hex from this URL (relay convention: <relay-origin>/fingerprint). */
  url?: URL | string;
}

export interface ConnectOptions {
  /** wss/https relay URL. http(s)-localhost or https only (secure context). */
  url: URL | string;
  /** Self-signed relay pinning. Omit for publicly-trusted certs. */
  fingerprint?: FingerprintSource;
  /**
   * Which MoQ draft(s) to advertise as WebTransport subprotocols (ALPN).
   * "auto" (default) advertises the full ordered list; pin a single draft to
   * force it. NEVER hardcoded to one wrong draft — moq-rs currently negotiates
   * DRAFT_14 wire but advertises ALPN; relay drives the choice.
   */
  draftPolicy?: MoqDraftPolicy;
  /** Passed through verbatim (allowPooling, congestionControl, extra protocols). */
  webTransport?: WebTransportOptions;
}

export interface MoqTransport {
  /** Ready WebTransport (wt.ready already awaited). */
  transport: WebTransport;
  /** Negotiated ALPN subprotocol string, or "" / undefined if none. */
  protocol: string | undefined;
}

/** Throws MoqTransportError on any failure (see error surfaces). */
export function connectMoqTransport(opts: ConnectOptions): Promise<MoqTransport>;
```

**Why this shape:**
- Returns a **raw ready WT + ALPN**, not a MoQ session (see Summary). Players
  keep their SETUP code; they just stop hand-rolling the connect prefix.
- `draftPolicy: "auto"` maps to the factory's canonical ordered ALPN list
  (single source of truth, replacing the 3 divergent lists). VERIFIED canonical
  candidates from code: `moqt-17, moqt-16, moqt-15, moq-00` (IETF) plus moq-lite
  ALPNs if hang must keep lite support — **open decision** whether the factory
  carries moq-lite ALPNs or hang passes them via `webTransport.protocols`.
- `fingerprint` accepts pre-decoded OR fetch-URL → covers all three players with
  one field. Hex normalization (strip `:`, validate `^[0-9a-fA-F]{64}$`,
  decode) lives in the factory once.

### Error surfaces (explicit — no silent fallbacks, per repo principles)

`MoqTransportError` with a `code` discriminant:
- `INSECURE_CONTEXT` — `!window.isSecureContext` (page is `file://`/`data:`/plain
  `http:` non-localhost). Throw *before* attempting connect with an actionable
  message. **NEW** vs today (none of the three check this).
- `WEBTRANSPORT_UNAVAILABLE` — `typeof window.WebTransport === "undefined"`
  (mirrors Shaka `:94`).
- `FINGERPRINT_FETCH_FAILED` — fetch non-ok / threw / non-hex body (today hang
  only `console.warn`s and proceeds — that masks the bug; factory should throw
  unless caller explicitly allows degraded connect).
- `FINGERPRINT_INVALID` — value not 32 bytes / not 64 hex chars.
- `CONNECT_FAILED` — `wt.ready` rejected (wrap the underlying `WebTransportError`;
  the Chrome-146 capsule/QUIC_ERROR 122 draft-skew break surfaces here — include
  the relay URL and advertised ALPN list in the message to make draft-skew
  diagnosable).

**Capsule/draft-mismatch note (document, don't fix here):** the known
Chrome-146 ↔ moq-rs break was a *relay-side* WebTransport-over-H3 draft
(`web-transport-proto` bump), not client-fixable. The factory cannot prevent it,
but its `CONNECT_FAILED` message should name the symptom so future readers
connect the dots. The factory must NOT hardcode a WT-over-H3 draft — that is the
browser's/relay's concern, not a `serverCertificateHashes` option.

## Package shape & import-map integration

**Location:** `packages/moq-transport/` (sibling to `ssm-transport`, `shared`).
TS source, `tsc` build to `dist/`, browser-targeted, **zero runtime deps**
(uses only `WebTransport`, `fetch`, `crypto`-free hex decode). Hex helper inlined
(matches `ssm-transport`'s "inline to avoid cross-package dep" pattern,
`ssm-transport/src/abr-controller.ts:16`).

**Import-map externalization — VERIFIED pattern, follow it.** Per project
CLAUDE.md "Import-map externals", `@blockcast/*` is loaded via browser import map,
not bundled:
- `hang-mmt-fec/js/watch/vite.config.ts:40` —
  `external: [..., /^@blockcast\//]`.
- moqtail + hang vite configs alias `@blockcast/mmt-fec`, `@blockcast/mmt-container`
  (`moqtail-private/apps/client-js/vite.config.ts:33–45`,
  `hang-mmt-fec/js/watch/vite.config.ts:11–24`).
- The player HTML carries the import map: `packages/moq-server/public/players/moq-lib.html` (VERIFIED contains `importmap`).

**Therefore:** `@blockcast/moq-transport` must be:
- listed in the `external: [/^@blockcast\//]` set (already matched by the regex —
  no vite change needed for hang),
- added to the `<script type="importmap">` in the player HTML(s) pointing at the
  built ESM,
- shipped as ESM with a stable entry so the import-map URL resolves.
- **NOT** bundled into `public/js` / `public/client-js/assets` (those are build
  output; per CLAUDE.md never edited directly). The build chain
  (`build:players`: libmmt → moqtail → moq-lib) does not need a new stage if the
  factory is import-map-external; if any player chooses to inline it, add a
  `build:moq-transport` stage before `build:moqtail`.

**Shaka caveat (VERIFIED divergence):** Shaka is **Closure-compiled**
(`goog.provide`/`goog.require`), not ESM/vite. It cannot consume an
`@blockcast/*` import-map module the way hang/moqtail do. Two options:
- **(A)** Shaka adopts the factory only via its existing `transportFactory`
  config seam (`msf_transport.js:92`, `player_configuration.js:764`): the *app
  embedding Shaka* passes a `transportFactory` that calls
  `connectMoqTransport().then(r => r.transport)`. Zero change to Shaka core.
  Preferred — the seam already exists for exactly this.
- **(B)** Vendor the factory into Shaka's Closure tree. Rejected: duplicates
  source, breaks single-source-of-truth.

## Per-player migration plan (behavior-preserving)

General rule: replace only the lines from "build options" through
"`await wt.ready`"; keep each player's SETUP/version-branch code unchanged.

### Shaka (`lib/msf/msf_transport.js`)
- **No core edit.** Provide factory at the **app** layer via
  `msf.transportFactory = (uri, options) => connectMoqTransport({ url: uri, fingerprint: { value }, webTransport: options }).then(r => r.transport)`.
- The factory takes over `serverCertificateHashes` + ALPN; Shaka's `connect()`
  still passes `options.protocols` today (`:119–126`), so to avoid double-owning
  ALPN, the app-supplied factory should let the factory's `draftPolicy` win and
  ignore/merge `options.protocols`. **Open decision:** who owns ALPN when both
  Shaka config and factory have an opinion. Cleanest: Shaka stops setting
  `options.protocols`, factory owns it — but that IS a core edit. If "no core
  edit" is required, factory must merge.
- Behavior preserved: same hashes, same `wt.ready`, same `.protocol` read at
  `:148`.

### hang / moq-lib (`js/lite/src/connection/connect.ts`)
- Replace `connectWebTransport()` body (`:388–429`) with a call into the factory:
  `connectMoqTransport({ url, fingerprint: props.fingerprintUrl ? {url: props.fingerprintUrl} : undefined, webTransport: webtransportOptions })`.
- **Behavior to preserve carefully:** the `http://`→`/certificate.sha256`
  auto-fetch + `https://` rewrite (`:404–427`) and the moq-lite ALPN entries
  (`:398`). The factory's `/fingerprint` convention differs from
  `/certificate.sha256`; migration must either (a) make the factory's fetch path
  configurable to `/certificate.sha256` for backward-compat, or (b) confirm the
  relay now serves `/fingerprint` and retire the old path (cross-check relay).
  **Open.** The WebSocket-fallback race (`:106–137`) stays in hang — out of
  factory scope.

### moqtail (`apps/client-js/src/lib/player.ts` + `libs/moqtail-ts/client.ts`)
- App `initialize()` (`player.ts:690`) currently hand-decodes hex
  (`:702–706`) into `transportOptions`. Replace with factory: pass
  `fingerprint: { value: this.#options.fingerprint }` and feed the resulting
  `transport` into `MOQtailClient.new({ ..., transportOptions: { ... } })`.
- **But** `MOQtailClient.new()` itself constructs `new WebTransport`
  (`client.ts:465`) and pushes `SUPPORTED_VERSIONS`. To use the factory without
  forking moqtail-ts, the cleanest path is a new `MOQtailClient.fromTransport(wt)`
  overload (lib change) OR keep moqtail self-connecting and only adopt the
  factory's hex/secure-context helpers. **Open decision** — moqtail-ts is a
  vendored fork (`moqtail-private`), so a small `fromTransport` addition is
  feasible but is the largest of the three migrations.

**Migration ordering:** hang first (cleanest seam, already import-map-external),
then moqtail (needs `fromTransport`), Shaka last (app-layer only, no core risk).

## Risks & open decisions

1. **Name collision (HIGH).** `@blockcast/transport` = existing `ssm-transport`.
   Must pick a new name (`@blockcast/moq-transport`). Surfaced above — needs owner
   sign-off before any code.
2. **Raw-WT vs session-factory (HIGH, decided here as raw-WT).** If a future
   reviewer wants a session-level factory, that is a much larger MoQ-session
   unification effort (3 incompatible session layers) — out of T1.6.
3. **ALPN ownership / draft skew (HIGH).** moq-rs negotiates DRAFT_14 wire but
   players advertise `moqt-16`/lite ALPNs. Factory's `draftPolicy:"auto"` must
   advertise a list the relay accepts; pinning a single wrong draft breaks
   connect. Do NOT hardcode. The Chrome-146 capsule break is relay-side and
   undetectable client-side beyond a clearer error message.
4. **Fingerprint endpoint mismatch (MED).** hang legacy `/certificate.sha256`
   vs relay `/fingerprint`. Migration must reconcile — verify what moq-rs relay
   serves (READ-ONLY context says `/fingerprint` hex over h2 on the WT addr).
   ASSUMED `/fingerprint`; VERIFY against relay before retiring the old path.
5. **Fingerprint fetch CORS/TLS (MED).** Fetching `/fingerprint` from the relay
   origin may be cross-origin from the player page; relay must send permissive
   CORS, or the page must be same-origin. ASSUMED same-origin in current E2E
   (test harness injects `window.__moqFingerprint` instead of fetching) —
   production cross-origin fetch is UNVERIFIED.
6. **Shaka Closure-compile incompatibility (MED).** Cannot import an ESM
   `@blockcast/*` module; must use `transportFactory` seam at app layer. VERIFIED
   the seam exists.
7. **ECDSA ≤14-day cert constraint (LOW, doc-only).** `serverCertificateHashes`
   requires ECDSA + ≤14-day validity (IWA cert gen already uses `validDays: 14`,
   VERIFIED `iwa/.../webtransport-server.ts:717`). Factory doesn't generate
   certs; just document the pin requirement.
8. **Import-map vs bundling (LOW).** Following the `/^@blockcast\//` external
   pattern means no bundle stage; if a player inlines instead, a
   `build:moq-transport` stage must precede `build:moqtail`.

   32-byte check), error-code mapping (insecure context, missing WT, bad
