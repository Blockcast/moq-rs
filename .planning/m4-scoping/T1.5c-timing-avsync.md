## Summary

T1.5c owns three things, which split cleanly into two tiers by whether they need
audio:

- **TIER 1 (1.B SHIPPED; 1.A doable now *pending OD-1*, video-only):** the two
  timing edges T1.2 explicitly deferred —
  1. a **reorder / jitter buffer** on the video path so playback is seamless
     under heavy frame reorder (the *timeline invariant* is already unit-proven;
     seamless *playback* is not), and
  2. a **robust NTP-short ~18.2 h wrap-boundary unwrap** that survives reorder
     across the boundary (currently a one-way epoch latch that mis-offsets).
- **TIER 2 (BLOCKED-ON-B):** the actual **A/V-sync core** — a single shared sync
  reference both tracks pace against, following moq-lib's `Sync` / moqtail's
  `MediaSync`. This **cannot be built or tested this pass** because Sub-project B
  (publisher audio: an FFmpeg `moq_mmt` audio MMTP track + catalog) does not
  exist yet. There is no second track to sync against. Design it on paper; gate
  the build.

The single most important fact for TIER 1: **the jitter window must be derived
from the catalog's FEC `interleaveDepth` + `sourceSymbols` (block span), the same
derivation moqtail's `MediaSync` already uses** — NOT a hardcoded buffer. But the
current Shaka `msfCatalog.Track` typedef carries **no FEC fields** and the T1.2
stream is **FEC-off**, so a real derivation has a precondition that must be
resolved first (see TIER 1 §A, open decision OD-1). This is the crux of TIER 1.

## Current timing model (VERIFIED in code, file:line)

How a video frame's presentation time is derived today, end to end:

1. **Wire timestamp (publisher ground truth).** The FFmpeg `moq_mmt` muxer writes
   a per-sample **NTP-short** (16.16 fixed-point) MMTP `timestamp` into the 12-byte
   MMTP header of **every** packet, **constant across all fragments of one MFU**,
   **distinct + monotonic per frame** (`frame0=0x0`, `frame1=0x1111`, …).
   VERIFIED empirically 2026-05-29 — see memory
   `ffmpeg-moq-mmt-multicast-wire-format.md`. The stream is currently **FEC-off**
   (VERIFIED: `m4-t1.2-video-playback-design.md:28`, D11).

2. **Parse + reassemble.** `MMTPParser.parse()` extracts `timestamp` /`rapFlag`;
   `MmtpTrackProcessor.process()`
   (`shaka-player/lib/msf/mmtp_track_processor.js:57-169`) strips the 14-byte
   sample-based MFU sub-header on the first fragment of a timed MFU
   (`:118-125`) and feeds `MfuReassembler`.
   `MfuReassembler.addFragment()`
   (`shaka-player/lib/msf/mfu_reassembler.js:83-149`) groups fragments by
   `(groupId, subgroupId)`, **sorts/dedupes by `objectId`** (`:129-146`), and
   emits a complete MFU. The reassembled MFU's `timestamp` is taken from the
   **first (lowest-objectId) fragment** (`:198-199`). **Memory is bounded by
   `maxBufferedMfus` with arrival-order eviction, no timer** (`:112-120`); the cap
   is `OBSERVE_FIRST_MAX_BUFFERED_MFUS_ = 256`
   (`shaka-player/lib/msf/msf_parser.js:1407`, passed at `:643-644`).
   **KEY:** this reorders *within* an MFU only. **Across complete MFUs / frames
   there is NO reorder buffer** — MFUs are emitted to the caller in *completion/
   arrival* order. (VERIFIED.)

3. **Timeline decode.** `processMmtpTrack_`
   (`shaka-player/lib/msf/msf_parser.js:566-768`) calls
   `MmtpTimeline.toPresentationSeconds(rec.timestamp)` (`:710`) to get a
   `startTime`, then builds a per-frame `SegmentReference`
   (`startTime → startTime + frameDuration`, `:711-720`).
   `frameDuration = 1 / track.framerate` from
   `mmtpFrameDuration_()` (`:537-544`) — **throws** if framerate absent
   (no fallback). `timescale = MmtpTransmuxer.VIDEO_TIMESCALE_` (`:586`).

4. **NTP-short unwrap (the wrap guard).** `MmtpTimeline`
   (`shaka-player/lib/msf/msf_parser.js:1425-1457`):
   - `raw = ntpShort >>> 0` (`:1443`).
   - **wrap guard:** `if (this.prevRaw_ !== null && this.prevRaw_ - raw >
     0x80000000) this.epochSeconds_ += 65536;` (`:1447-1449`). A backward jump of
     more than half the 32-bit range is treated as the ~18.2 h integer-seconds
     rollover; epoch advances by one `2^16 s` range.
   - `seconds = epochSeconds_ + raw / 65536` (`:1451`); first call sets
     `baseSeconds_`; return is `seconds - baseSeconds_` (`:1452-1455`).
   - `prevRaw_` is updated to the **last value seen on any call** (`:1450`),
     regardless of ordering.

5. **MSE wiring.** `SegmentReference` is merged into the `SegmentIndex`
   (`mergeAndEvict` once the timeline is locked, else `merge`, `:726-734`),
   notified to the `presentationTimeline_` (`:736-746`); for video-only the
   timeline `lockStartTime()`s on the first video segment (`:749-753`). MSE / the
   `MmtpTransmuxer` then CMAF-wrap the AVCC NAL and append. Eviction floor:
   `Math.min(reference.startTime - 2, getSegmentAvailabilityStart())` (`:729-730`)
   — the `- 2` is a 2-second look-back, **the one timing magic number in the
   path** (flagged under Risks).

**ASSUMED, not yet verified:** that MSE/Shaka's own buffering tolerates the
out-of-order *append* of `SegmentReference`s when frames arrive reordered. The
T1.2 invariant proof is purely at the `toPresentationSeconds` level
(`test/msf/msf_parser_unit.js:295-313`, "startTime is timestamp-derived under
reorder + loss"); it asserts the *startTime* is correct, not that playback is
glitch-free. That gap is exactly TIER 1 item 1.

## TIER 1 — video timing edges (doable now)

### A. Reorder / jitter buffer for the video path

**Problem (VERIFIED).** Frames arriving out of order get a correct `startTime`
(timestamp-derived) but are *appended* to MSE in arrival order. There is no stage
that holds a frame back to wait for an earlier-timestamp frame still in flight.
Under heavy reorder this produces non-monotonic appends / visible glitches even
though the timeline math is right. The reassembler reorders within an MFU only
(`mfu_reassembler.js:129-146`); nothing reorders across MFUs.

**Design: a small per-track reorder buffer between the reassembler and the
SegmentReference build.** It sits **inside `processMmtpTrack_`'s subscribe
callback** (`msf_parser.js:654-757`), *after* the first-RAP gate (`:703-708`) and
*before* the `SegmentReference` construction (`:710`). Mechanics:

- Buffer completed `mfu` records keyed by their decoded presentation time
  (`toPresentationSeconds(rec.timestamp)`), kept sorted.
- **Release rule (no timer, per repo rules):** release a held frame to the
  SegmentReference build when the highest *received* timestamp exceeds the held
  frame's timestamp by the **buffering window W** (i.e. we are confident no
  earlier frame is still coming). Eviction rides on arrival, exactly like
  `MfuReassembler` (`mfu_reassembler.js:112-120`) — this is the established
  no-timer pattern in this codebase.
- Interaction with first-RAP gate: the gate stays *upstream* of the buffer (we
  never buffer pre-RAP frames; gate logic at `:703-708` is unchanged).
- Interaction with eviction: once a frame is released it flows through the
  existing `mergeAndEvict` path (`:726-734`) untouched. The reorder buffer only
  changes *ordering and release timing* of appends, not the merge/evict logic.

**Where W comes from (the derivation — THE central rule).** Per CLAUDE.md
"Zero-copy, no defaults, no timers" and the existing precedent, **W is the FEC
block span derived from the catalog**, identical to moqtail's `MediaSync` buffer:

> `interleaveDepth (catalog) = D × (1000/fps) ms`  (the interleave WINDOW, ms)
> `blockSpan = (K − 1) × interleaveDepth`           (K = `sourceSymbols`)
> `W = codecJitter + 2 × blockSpan`

VERIFIED source of this exact formula:
`moqtail-private/apps/client-js/src/lib/player.ts:1640-1697` (audio buffer
derivation) and the `MediaSync` doc comment `:262` ("Buffer (jitter tolerance) is
derived from the catalog's FEC interleaveDepth"). The helper that reads the
catalog FEC config is `fecInterleaveMs()` (`player.ts:113-116`,
`interleaveDepthMs ?? interleaveDepth ?? interleaveMs`); `sourceSymbols` is K
(`player.ts:236`, `multi-track-wiring.ts:41`). moqtail **throws** if these are
absent (`player.ts:1665-1673`) and range-checks `interleaveDepth ≥ 33 ms`
(`:1678-1684`). The reorder window is the *interleave window* — NOT the block
span — and block span = `(K−1) × interleaveDepth` (CLAUDE.md "moq-lib"
anti-pattern note: `computeFecTimeout` / interleaveDepth ≠ block span).

**OPEN DECISION OD-1 (blocking the *real* derivation, NOT blocking the
plumbing).** The current Shaka catalog has **no FEC fields**: `msfCatalog.Track`
(`shaka-player/externs/msf_catalog.js:61-93`) carries `framerate`, `timescale`,
`targetLatency` — but **no `interleaveDepth`, no `sourceSymbols`, no
`getFecConfig`** (VERIFIED — contrast moqtail's `CMSFCatalog.getFecConfig`,
`endpoint-tracks.ts:29`). And the T1.2 stream is **FEC-off**
(`m4-t1.2-video-playback-design.md:28`). So *today* there is no `interleaveDepth`
to derive W from. Three options, pick one before implementing:

  - **OD-1a (recommended):** plumb FEC params into the Shaka MMTP catalog
    (publisher already emits them on the FEC-on path; this converges with
    **T1.FEC**). W then derives identically to moqtail. Clean, but couples T1.5c
    to T1.FEC landing the FEC-on catalog.
  - **OD-1b:** for the FEC-off stream, derive a *degenerate* W from `framerate`
    alone — e.g. a small integer number of frame-durations (`N / framerate`)
    where N is a catalog-supplied reorder depth. This still needs a catalog field
    (`reorderDepth` or reuse `targetLatency`, `msf_catalog.js:86`) to avoid a
    magic N — so it is **not** truly free of a catalog change.
  - **OD-1c:** gate the *whole* reorder buffer on FEC-on (W only ever derives from
    block span) and ship TIER 1 item 1 as **also blocked on T1.FEC**. Honest but
    leaves item 1 not-doable-now.

  **RESOLVED (2026-06-01) — OD-1a + a FEC-off fallback; reject pure OD-1c.**
  - **FEC-on:** `W` derives from a shared **catalog FEC descriptor**
    `{ fecProtected, interleaveDepthMs, sourceSymbols(K), algorithm }` that
    **T1.FEC (BLO-8645) owns and lands** (muxer already has the values:
    `DEFAULT_FEC_K=32`, `DEFAULT_FEC_DEPTH=30`, `fec_algorithm`). This issue
    *consumes* `interleaveDepthMs`+`K`.
    - **OPEN SUB-Q (pin before T1.FEC defines the descriptor — Ally #12):** the
      exact W formula depends on the UNIT of the interleave param. The muxer's
      `DEFAULT_FEC_DEPTH=30` is commented *"~1s protection at 30fps"* = **30
      frames**, NOT 30 ms — so a naïve `W=(K−1)·30ms` over-counts ~31× and would
      blow `targetLatency`. Resolve against moqtail's `MediaSync` impl whether the
      catalog field is depth-in-frames, inter-block spacing (ms), or the total
      interleave window (ms), and define `interleaveDepthMs` + the W formula
      accordingly. Do NOT ship a formula until this unit is pinned.
  - **FEC-off (today's stream):** no FEC interleave → derive a minimal W from the
    **existing `targetLatency`** field (`msf_catalog.js:86`). A (b)-style fallback.
  - **Net:** the reorder buffer is **never gated off** (reject pure OD-1c); W's
    *source* switches on `fecProtected`. Both paths throw if their field is absent
    (mirror `mmtpFrameDuration_`). **SEQUENCING: after BLO-8645's catalog work**
    (or split the descriptor into a tiny shared prerequisite). Posted on BLO-8647.

### B. Robust NTP-short ~18.2 h wrap-boundary unwrap (SHIPPED — see Status update)

> **SHIPPED** in shaka-player PR #6 (squash `62bef06f`). The design below is the
> as-built record; "Files to change" / "Ordered plan" read prospectively but are
> done. See the Status update at the end of this doc.

**Root cause (VERIFIED).** `MmtpTimeline.toPresentationSeconds`
(`msf_parser.js:1442-1456`) maintains a **one-way epoch latch** keyed on the
*last call's* raw value (`prevRaw_`, `:1450`) with a single threshold guard
(`prevRaw_ - raw > 0x80000000`, `:1447`). Two failure modes when reorder coincides
with the boundary:

  - **Mode 1 — late pre-wrap frame double-counts.** Once a frame at `raw ≈ 0x0000…`
    has advanced the epoch by 65536, a *reordered* frame from just *before* the
    boundary (`raw ≈ 0xFFFF…`) is computed as
    `epochSeconds_(65536) + 0xFFFF…/65536 ≈ 131071 s` instead of its true
    `≈ 65535 s` — off by a full `2^16 s` (~18.2 h).
  - **Mode 2 — spurious epoch advance.** A reordered frame arriving *before* the
    real wrap, whose `raw` is far enough below `prevRaw_` to exceed the
    `0x80000000` threshold, advances the epoch *early* and corrupts every
    subsequent frame until (if ever) it self-corrects.

The existing tests never exercise reorder *across* the boundary: the wrap test
(`test/msf/mmtp_transmuxer_unit.js:240-247`) is monotonic-forward only, and the
"small backward step" test (`:249-254`) stays far from the boundary. This is
exactly why the handoff calls it "threshold-guarded vs reorder … effectively
unreachable in practice but not robust" (`m4-t1.2-DONE-handoff.md`, Deferred).

**Robust design.** Make unwrap a **pure function of the raw value relative to a
stable epoch anchor**, not a function of call/arrival order. Standard
sequence-number-unwrap technique:

- Keep an epoch anchored on a *reference raw* (e.g. the base/first-accepted raw),
  not on `prevRaw_`.
- For each incoming `raw`, choose the epoch `e ∈ {epoch−1, epoch, epoch+1}` that
  minimizes `| (e·2^16 + raw/65536) − expectedSeconds |`, where `expectedSeconds`
  is a slowly-advancing estimate of the live timestamp (it can lean on the
  monotone wire fact: timestamps are per-frame monotone modulo reorder ≤ W). This
  makes a late pre-wrap frame resolve to the *near* epoch (correct) and a
  reordered frame never latches the epoch forward.
- The epoch only *commits* forward when a frame is confidently past the boundary
  (e.g. consistent with `expectedSeconds`), so a single reordered straggler can't
  move it. This composes naturally with the TIER 1.A reorder buffer: within the
  buffer window W, the unwrap sees a *locally sorted* stream, so the
  nearest-epoch choice is unambiguous.

This is a **few-line change** to `toPresentationSeconds` (`:1442-1456`) plus the
constructor's tracked fields (`:1426-1433`); resist building a general clock.

**Files to change (TIER 1.B):**
- `shaka-player/lib/msf/msf_parser.js` — `MmtpTimeline` constructor (`:1426-1433`)
  + `toPresentationSeconds` (`:1442-1456`) only.

**Ordered plan (TIER 1.B):**
1. Add the failing tests first (TDD): reorder a pre-boundary frame to arrive
   after a post-boundary frame (Mode 1); inject a backward straggler before the
   wrap (Mode 2). Both should currently mis-offset by ~65536 s.
2. Replace the one-way latch with nearest-epoch resolution against an anchored
   estimate.
3. Confirm the three existing tests (`mmtp_transmuxer_unit.js:231-254`) still
   pass unchanged (no regression to monotonic + small-reorder behavior).

**Note on coupling:** TIER 1.B is *cleaner* once 1.A exists (the buffer hands it a
locally-sorted stream), but 1.B is independently correct and can land first or
alone. 1.A does **not** depend on 1.B.

## TIER 2 — A/V sync core (BLOCKED-ON-B)

**BLOCKED-ON-B. Cannot build or test this pass — there is no audio track to sync
against.** Sub-project B (publisher audio) does not exist. This section is paper
design only, following the mandated existing pattern (CLAUDE.md: "do NOT invent a
new sync model" — adopt moq-lib `Sync` / moqtail `MediaSync").

### The pattern to follow (VERIFIED references)

- **moq-lib `Sync`** (`moq-lib/js/watch/src/sync.ts`): a single shared
  `#reference` = earliest wall-time seen relative to a frame's media timestamp
  (`received()`, `:114-156`); a derived `#buffer` = `jitter + max(audio, video)`
  (`#runBuffer`, `:101-110`); both tracks call `await sync.wait(timestamp)`
  (`:163-188`) which sleeps until the shared reference + buffer says it's time.
  One reference, both tracks wait against it → no per-track drift.
- **moqtail `MediaSync`** (`moqtail-private/apps/client-js/src/lib/player.ts:265-309`):
  the same idea adapted — `#anchorWallMs`/`#anchorMediaMs` anchored on the
  earliest frame across *both* tracks (`received()`, `:274-289`; "Update anchor if
  this frame is earlier (e.g. audio arrives before video)"), `wait()` sleeps
  `elapsed − wallElapsed + bufferMs` (`:292-303`). Constructed once with a single
  `syncBufferMs` and shared by both pumps
  (`#sync = new MediaSync(syncBufferMs)`, `:1778`; both pumps pace against it,
  `:413`, `:1119`).

### Sketch for MMTP-in-Shaka

When B exists, a shared `MmtpSync` (one per presentation, modeled on `MediaSync`)
would be constructed in the MSF parser and shared by the video `processMmtpTrack_`
path and the (new) audio path. Both convert their NTP-short MMTP timestamp via the
**same** epoch (so the two tracks land on one media-time axis) and register/wait
against the one reference. The TIER 1.A reorder buffer per track feeds frames into
the shared sync in timestamp order; the sync's `bufferMs` is the **same FEC
block-span derivation** as TIER 1.A's W (`player.ts:1770-1779`,
`syncBufferMs = fecInterleaveMs(aFec) ?? fecInterleaveMs(vFec)`).

### The B dependency — what B MUST expose (cross-referenced)

For a *shared* reference to be meaningful, audio and video timestamps must be on a
**single common clock with a shared epoch**. From the publisher ground truth
(`ffmpeg-moq-mmt-multicast-wire-format.md`): video MMTP `timestamp =
us_to_ntp_short(base_pts + i·sample_dur)`. B must guarantee the **audio MMTP
timestamps share that same `base_pts` epoch and NTP-short encoding** — i.e. audio
sample 0 and video frame 0 are expressed against one wall/media origin, not two
independent per-track origins. If B emits audio on an independent epoch, the shared
reference is meaningless and A/V will drift regardless of the sync class.

**THE SINGLE MOST IMPORTANT THING B MUST EXPOSE:** a **shared presentation clock /
epoch common to the audio and video MMTP tracks** — concretely, audio and video
NTP-short timestamps derived from the *same* `base_pts` origin (one media-time
axis), so a single `MmtpSync` reference aligns both. Secondary: the catalog audio
track must carry the **FEC params** (`interleaveDepth` ms + `sourceSymbols`) so the
sync `bufferMs` derives without a magic number — same OD-1 gap as TIER 1, now
mandatory for audio (moqtail throws otherwise, `player.ts:1665-1673`).

**Every TIER-2 item is BLOCKED-ON-B.** Do not implement; revisit when B lands
(and ideally after Sub-project C — receiver audio — per the umbrella in
`m4-t1.2-DONE-handoff.md`, "Next session targets").

## Risks & open decisions

- **OD-1 (TIER 1, blocking the real W derivation):** the Shaka catalog has no FEC
  fields and the stream is FEC-off. Choose OD-1a (plumb FEC catalog, converge with
  T1.FEC — recommended) / OD-1b (framerate+`reorderDepth` field) / OD-1c (gate on
  FEC-on). **Owner decision needed before TIER 1.A implementation.** TIER 1.B is
  unaffected.
- **R1 — eviction `- 2` magic look-back** (`msf_parser.js:729-730`): a 2-second
  literal in the eviction floor. Not introduced by T1.5c, but a reorder buffer of
  window W interacts with it; if W approaches 2 s the look-back could evict a frame
  the buffer still intends to release. Verify W ≪ 2 s or derive the look-back too.
- **R2 — MSE out-of-order append tolerance (ASSUMED):** the T1.2 proof is at the
  timeline-math layer only; whether MSE/Shaka append-order tolerance is the real
  source of reorder glitches is *assumed*, not measured. The reorder buffer
  presumes append order is the problem — confirm with an E2E heavy-reorder repro
  *before* building, so we fix the right layer (systematic-debugging rule).
- **R3 — wrap unwrap is genuinely ~unreachable** (~18.2 h): TIER 1.B is
  correctness-hardening, low practical urgency. Worth doing because it's a few
  lines + a test, but should not block 1.A.
- **R4 — simplicity:** resist building a general "media clock" abstraction in
  TIER 1. The reorder buffer is a sorted hold with arrival-driven release; the
  unwrap fix is nearest-epoch resolution. Both are small. The shared-clock
  abstraction belongs in TIER 2, modeled on `MediaSync`, not invented now.

## Status update + follow-ups (2026-06-01)

**TIER 1.B SHIPPED.** The NTP-short wrap unwrap was implemented as a nearest-epoch
resolution against a monotonic anchor (replacing the one-way `prevRaw_`/`epochSeconds_`
latch). Merged to `blo-4020-m4-t1-shaka-mmtp` via shaka-player PR #6 (squash commit
`62bef06f`), Ally-clean (0 Crit / 0 Imp). Unit: `shaka.msf` 149/149 (+2 reorder-across-wrap
specs). The design-sketch ASSUMED above is now test-pinned.

**Follow-ups raised by Ally on PR #6 (both non-blocking, deferred here on purpose):**

1. **Anchor poisonability** (`lib/msf/msf_parser.js`, the `seconds > anchorSeconds_`
   advance). A single spurious large-forward NTP-short value latches the anchor
   permanently forward; subsequent genuine frames >~2^15 s behind it resolve to the
