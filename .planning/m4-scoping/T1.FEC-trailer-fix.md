# T1.FEC — FEC Trailer Fix (Design Decision D11)

**Milestone:** M.4 MMTP-over-MoQ
**Status:** SCOPING (this doc plans; it does not implement)
**Author:** scoping pass, 2026-06-01
**Repos investigated (read-only):** FFmpeg fork, moq-rs (`moq-pub-mmtp`), libmmt (+ `codex/mmtp-fec-trailer-timeout`), shaka-player, fec-raptorq.

---

## Summary

The FFmpeg `moq_mmt` muxer appends the 4-byte **Source FEC Payload ID (SS_ID)**
as a **TRAILER** — the last 4 bytes of the MMTP packet, written *after* the MFU
media data, past the declared MPU `payload_length`. Two downstream parsers
instead model `fec_type==1` as a 4-byte **PREFIX** sitting *between* the 12-byte
MMTP header and the 8-byte MPU header:

- moq-rs's vendored Rust `mmt-core` (`MmtpHeaderExt::read_from`/`write_to`), and
- Shaka's `lib/msf/mmtp_parser.js`.

This is a real, triple-confirmed parser-parity gap (see table below). It is
**benign today only** because the production stream runs **FEC-OFF**
(`fec_enable` defaults to `0`), so `fec_type` is always `0`, the prefix branch
is never entered, and `MpuHeader.payload_length` bounds every parse. The moment
`fec_enable=1` is turned on, the two prefix parsers will mis-slice the packet:
they would consume the first 4 media bytes as a "prefix SS_ID" and shift the MPU
header read by 4 bytes, corrupting `payload_length`, `mpu_sequence`, and the MFU
boundary.

The parallel libmmt effort on `codex/mmtp-fec-trailer-timeout` has **already
chosen the correct contract: SS_ID is a TRAILER**, bounded by comparing the
on-wire byte count against the declared payload end (+4). **However, that branch
fixed only the TypeScript path** (`packages/mmt-base/src/mmtp.ts`); the **Rust
`mmt-core/src/header.rs` is byte-identical between `main` and the codex branch**
and still serializes/parses SS_ID as a prefix. So the Rust + Shaka fixes are
genuinely net-new work that must port libmmt's TS trailer contract.

**Distinguishing verified vs assumed** is called out inline with `[VERIFIED]`
and `[ASSUMED]` tags throughout.

### Corroborating detail `[VERIFIED]`

- **Muxer doc-comment confirms trailer intent**: `moqenc_mmt.c:1115-1119`
  (`build_mmtp_header` helper) — *"Source (fec_type=1): 4-byte Source FEC
  Payload ID appended."* "Appended" = trailer.
- **Muxer has a contradictory `#if 0` dead function**: `moqenc_mmt.c:1199-1203`
  `write_source_fec_payload_id()` writes `block_id(4) + symbol_id(4) = 8 bytes`.
  It is `#if 0` / unused and is **not** the live path. The live path (`:4517`)
  writes a single flat 4-byte `ss_id = fec_block_id * stream_k + fec_symbol_id`.
  → **Dead code; flag for the libmmt/muxer owner but do not let it confuse the
  contract. 4-byte flat SS_ID trailer is authoritative.**
- **Rust `mmtp_parse.rs` (the moq-pub-mmtp publisher's actual parse entry) does
  NOT itself model a prefix** — `moq-rs/moq-pub-mmtp/src/mmtp_parse.rs:62-77`
  calls `MmtpHeader::read_from` then `MpuHeader::read_from` and **never touches
  `SourceFecPayloadId`**. The prefix model lives one layer down in the vendored
  `mmt-core::MmtpHeaderExt`, which `mmtp_parse.rs` does not currently invoke.
  `[VERIFIED]` `MmtpHeaderExt` is referenced in moq-rs **only** by the test
  vector generator `examples/mmtp_packet_vectors.rs:26,105` and the vendored
  crate itself — i.e. it is **not on the publisher hot path today**. This
  narrows the Rust blast radius substantially (see Risks).
- **The example vector generator encodes the bug**:
  `moq-rs/moq-pub-mmtp/examples/mmtp_packet_vectors.rs:102-129` — its
  `fec_vector()` builds the packet as `ext.write_to()` (MMTP + prefix SS_ID)
  then `mpu.write_to()` then payload, and its docstring literally says SS_ID
  *"sits between the 12-byte MMTP header and the 8-byte MPU header"*
  (`:103-104`, `:271`). This file must be corrected as part of the fix or it
  will keep minting prefix-layout golden vectors.

## Spec authority (ISO/IEC 23008-1 §C.5.2)

ISO/IEC 23008-1:2023 (MMT) Annex C defines the AL-FEC payload IDs. `[ASSUMED
from in-repo citations — the standard text is not in any of these repos; the
authority below is reconstructed from the consistent §C.5.2 references in
`moqenc_mmt.c:103`, `mmt-core/src/header.rs:255-259`, and
`packages/mmt-base/src/mmtp.ts:253`.]`

- **`source_FEC_payload_ID`** (the `fec_type==1`/"with source payload ID" case)
  is the **AL-FEC source-symbol identifier appended to a source packet**. Per
  the in-repo citations it is carried as a **trailer past the MMTP payload**,
  not a header field inserted before the payload. `moqenc_mmt.c:122` is explicit
  that non-FEC-aware receivers treat the trailer as *"extra bytes past the
  declared MPU `payload_length`"* and keep working — this only holds if the ID
  is a **trailer** bounded by `payload_length`. A prefix would corrupt the MPU
  header for those receivers. **→ Trailer is the spec-correct placement.**
  `[VERIFIED against repo invariants; ASSUMED against the ISO text itself.]`

- **Size & breakdown — 4 bytes, single `SS_ID`** (NOT an SBN/ESI split on the
  wire). All three repos agree the on-wire field is one 32-bit `SS_ID`:
  - `mmt-core/src/header.rs:256-289`: `SS_ID (32 bits)`, with derivation
    `SBN = floor(SS_ID / K)`, `ESI = SS_ID % K` computed *by the receiver* for
    constant K. `SourceFecPayloadId::SIZE = 4`.
  - `moqenc_mmt.c:4515-4518`: `ss_id = fec_block_id * stream_k + fec_symbol_id`
    (the inverse encode), written as one `AV_WB32`. `[VERIFIED]`
  - i.e. **SBN/ESI are a logical decomposition of the single 4-byte SS_ID, not
    two separate wire fields.** The fec-raptorq codec consumes the derived
    `(SBN, ESI)` — see fec-raptorq context below.

- **Repair packets are a different, larger ID** — **do not conflate.** The
  *repair* FEC payload ID (`fec_type==2`, `§C.5.3`) is **13 bytes**:
  `SS_Start(4) + RSB_length(3) + RS_ID(3) + SSB_length(3)`
  (`moqenc_mmt.c:104`, `REPAIR_FEC_PAYLOAD_ID_SIZE`; written at `:1765-1789`).
  Repair packets carry **no** source FEC payload ID. The T1.FEC fix is scoped to
  the **4-byte source trailer only**; the repair-side layout is already a
  distinct code path and is out of scope here. `[VERIFIED]`

### fec-raptorq context (what SS_ID carries)
`[VERIFIED]` `fec-raptorq/` is the RaptorQ codec. The on-wire `SS_ID` is the
flat source-symbol sequence number; the receiver derives `(SBN, ESI)` =
`(SS_ID/K, SS_ID%K)` (constant-K) to address the RaptorQ block and symbol slot.
The trailer must therefore be parsed and *stripped from the media bytes* before
fMP4 reassembly, and the parsed `ss_id` handed to the FEC layer. The TS branch
already wires this via `MmtpContainerParser.onSourceSymbol`
(`libmmt` commit `929e5b0`).

## libmmt convergence contract

**Branch:** `codex/mmtp-fec-trailer-timeout` (libmmt monorepo). `[VERIFIED]`
HEAD = `929e5b0`; the two FEC-trailer commits are:
- `4f4f549 fix(container): strip relay-preserved FEC payload IDs`
- `929e5b0 feat(container): MmtpContainerParser.onSourceSymbol callback`

**Scope of the branch `[VERIFIED]`:** TypeScript only —
`packages/mmt-base/src/mmtp.ts`, `packages/container/src/*`. The Rust
`mmt-core/src/header.rs` is **byte-identical** between `main` and the codex
branch (`git diff main codex/... -- mmt-core/src/header.rs` is empty). **The
Rust prefix serializer is therefore unfixed upstream and is net-new T1.FEC work.**

**The chosen contract (port this to Rust + Shaka):**

1. **Where the FEC id lives:** **TRAILER** — the final 4 bytes of the MMTP
   packet, after the MFU media data, past MPU `payload_length`. NOT a prefix.

2. **How length is computed:** `payloadEnd = payload.length - 4` when the
   trailer is present (`mmtp.ts:256`). Media data ends 4 bytes early; the last 4
   are the big-endian `SS_ID`.

3. **How the parser bounds it (`hasSourceFecPayloadId`, `mmtp.ts:374-410`):**
   - Reject if packet `< MMTP_HEADER_SIZE + MPU_HEADER_SIZE + 4`.
   - Must be a `PACKET_TYPE_MPU` packet.
   - `header.fecType === 1` → **true** (trust the header bit).
   - `header.fecType !== 0` → false (repair etc. handled elsewhere).
   - `header.fecType === 0` (relay reset the bit but may have preserved the
     trailer): require `fragmentType === 2` (Mfu) and then bound by length:
     compute the two plausible payload ends and accept iff the on-wire
     `payloadBytes === end + 4`:
     - `endWhenLengthIncludesDu  = MPU_HEADER_SIZE + payloadLength`
     - `endWhenLengthIsMediaOnly = MPU_HEADER_SIZE + (mfuDuHeader? mfuHeaderSize:0) + payloadLength`
   - **`forceFecPayloadId` catalog override** (`mmtp.ts:381`,
     `mmtp-container-parser.ts` `ParseHeaderOptions`): when the track is known
     FEC-protected from the catalog, force the trailer path even if a relay
     zeroed `fec_type`. This is the "catalog-derived everything" pattern — the
     header bit is advisory, the catalog is authoritative.

4. **Relay caveat the branch encodes:** relays may forward source packets with
   `fec_type` rewritten to `0` while preserving the trailing SS_ID. Detection
   must not rely on the header bit alone. (Comment `mmt.ts:276-278` + commit
   `4f4f549` message.)

**Convergence requirement:** the Rust `mmt-core` and Shaka changes MUST adopt
this exact trailer + length-bounded + `forceFecId` model. Constants
(`MMTP_HEADER_SIZE=12`, `MPU_HEADER_SIZE=8`, `SS_ID size=4`) already match
across all repos `[VERIFIED]`.

### Phase A — Rust `mmt-core` (the prefix serializer)

The vendored copy at `moq-rs/moq-pub-mmtp/vendor/mmt-core/` is pinned to libmmt
`929e5b0` (`vendor/mmt-core/VENDOR.md`). **Fix upstream libmmt first, then
re-vendor** — editing only the vendored copy would diverge from the pin and the
next `VENDOR.md` refresh would silently revert it.

1. **libmmt `mmt-core/src/header.rs`** — change `MmtpHeaderExt`:
   - `write_to` (`:388`): write `base` header, **then MPU header + payload, then
     the SS_ID trailer** — i.e. SS_ID must no longer be emitted directly after
     the base header. (Likely means `MmtpHeaderExt` stops owning the
     serialization order and instead exposes a `trailer_bytes()`/append helper,
     since the MPU header + payload are written by the caller. **Design choice
     to confirm with libmmt owner — see Risks/Open Questions.**)
   - `read_from` (`:399`): stop reading SS_ID before the MPU header. Add a
     trailer-aware decode that mirrors TS `hasSourceFecPayloadId`: given the full
     packet + MPU `payload_length`, bound and slice the trailing 4 bytes.
   - Keep `SourceFecPayloadId` struct, `SIZE=4`, `sbn()/esi()` as-is `[VERIFIED
     correct]`.
   - Update the doc comment `:259` (already says "appended after MMTP payload" —
     correct intent; the code contradicts it).
2. **libmmt: add Rust unit tests** mirroring the TS RED/GREEN (below).
3. **Re-vendor** into moq-rs per `VENDOR.md` refresh procedure; bump pinned
   commit + `Vendored on` date.

### Phase B — moq-rs test vector generator + parse path

4. **`moq-rs/moq-pub-mmtp/examples/mmtp_packet_vectors.rs`** — fix `fec_vector()`
   (`:102-129`) to emit the **trailer** layout (write MPU header + payload, then
   SS_ID), and correct the misleading docstrings (`:103-104`, `:271`,
   `:7`, `:17`).
5. **`moq-rs/moq-pub-mmtp/src/mmtp_parse.rs`** — `route()` currently ignores
   SS_ID entirely (`[VERIFIED]` it only reads MMTP + MPU headers and bounds by

   trailer-bounded extraction here.** If routing genuinely only needs
   `packet_id`/`mpu_sequence`/`timestamp` (today's behavior), `route()` may stay
   header-only and the fix is purely "don't mis-model the prefix anywhere." →
   **Open question for the publisher owner (see Risks).**

### Phase C — Shaka

6. **`shaka-player/lib/msf/mmtp_parser.js`** — move SS_ID extraction from prefix

   - Read MPU header at `offset = 12` (no longer 16).
   - After computing the payload slice, port `hasSourceFecPayloadId` + the

7. **`shaka-player/lib/msf/mmtp_track_processor.js` / transmuxer** — verify the

### Phase D — shared golden vectors

8. **`shaka-player/test/test/assets/mmtp_packet_vectors.json`** is generated from

## Risks (keep FEC-OFF green)

1. **FEC-OFF path must not change behavior. `[VERIFIED low risk]`** All three
   parsers gate the SS_ID branch on `fec_type>=1` (or the new length+catalog
   detection, which requires a present trailer). On FEC-OFF every packet is
   `fec_type==0` with no trailer; the new `hasSourceFecPayloadId` returns false
   (`forceFecId` defaults off). The existing `moq_mmt_capture.json` replay test
   is the guard — **it must stay green untouched.** Watch the relay-`fec_type==0`
   detection heuristic: its length check (`payloadBytes === end + 4`) could in
   principle false-positive on a FEC-OFF packet whose media happens to be exactly
   4 bytes longer than `payloadLength`. The TS branch mitigates by also requiring
   `fragmentType===2` and by defaulting `forceFecId` off; **the Rust/Shaka ports
   must preserve both guards.** `[ASSUMED]` a pure FEC-OFF stream never trips this
   — validate by running the FEC-OFF replay after the change.

2. **Two payloadLength conventions. `[VERIFIED in TS]`** `hasSourceFecPayloadId`
   accepts SS_ID under *either* "payloadLength includes MFU DU header" or
   "media-only" convention (`mmtp.ts:399-409`). The Rust/Shaka ports must
   replicate both branches or they'll mis-bound on whichever convention the
   muxer emits. Confirm which convention `moqenc_mmt.c` actually uses for FEC-ON
   (the muxer sets `payload_length = video data bytes (excludes DU header)` per
   `mmtp.ts:262` comment — but verify against the FEC-ON capture).

3. **Mapping-B subgrouping interaction. `[VERIFIED relevant]`** Mapping B keys
   the per-MFU subgroup off the **MMTP timestamp** (`mmtp_parse.rs:38-44`,
   present on every fragment), NOT off SS_ID. The trailer fix changes *media
   payload length* (−4 bytes) and adds an `ss_id`, but does **not** touch the
   timestamp or `mpu_sequence` that drive subgroup/group assignment. → **No
   expected interaction**, but the FEC-ON replay test should assert subgroup
   bucketing is unchanged vs the FEC-OFF capture's shape.

4. **`route()` may not need SS_ID at all. `[VERIFIED today / OPEN for FEC-ON]`**
   The publisher's `mmtp_parse::route()` is header-only today and the prefix
   model is *not* on its hot path (only the vendored crate + test generator use
   `MmtpHeaderExt`). If FEC routing/repair association needs SS_ID at the
   publisher, Phase B step 5 adds extraction; if not, the publisher fix is
   limited to not minting prefix vectors. **Open question — confirm with the
   moq-pub-mmtp owner before sizing Phase B.**

5. **`MmtpHeaderExt` serialization-order ownership. `[OPEN]`** Today
   `MmtpHeaderExt::write_to` writes base+SS_ID as a unit. A correct trailer means
   SS_ID is emitted *after* MPU header + payload, which `MmtpHeaderExt` does not
   own. The clean fix likely demotes `MmtpHeaderExt` to "base header + a
   `trailer()` accessor" and lets the packet assembler append the trailer last —
   an API shape change. **Confirm the desired API with the libmmt owner so the
   re-vendor stays a clean cherry-pick.**

6. **Dead `#if 0` 8-byte writer in the muxer. `[VERIFIED]`**
   `moqenc_mmt.c:1199` `write_source_fec_payload_id` (block+symbol, 8 bytes) is
   unused and contradicts the live 4-byte flat-SS_ID trailer. Not load-bearing,
   but flag it for deletion/clarification to the muxer owner so it doesn't
   re-confuse a future implementer.

7. **Vendor drift. `[VERIFIED process risk]`** Fixing only the moq-rs vendored
   copy without fixing upstream libmmt `mmt-core` means the next `VENDOR.md`
   refresh reverts the fix. Order matters: **libmmt first, then re-vendor.**
